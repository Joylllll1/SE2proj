## Context

当前 NJU 树洞前端已完成基础功能开发，包括帖子发布、评论、点赞、收藏、活动申请审核、举报处理等核心流程。但在用户交互体验和管理效率方面存在以下问题：

- 活动时间输入为纯文本，格式不统一，展示体验差
- 活动分类缺少重要的校园场景类型（答辩、校招等）
- 优质活动内容无法在首页获得曝光
- 管理员审核举报时无法查看帖子完整内容
- 收藏功能过于简单，无法分类管理
- 活动不支持收藏，用户后续查找不便

技术约束：
- 纯前端 mock 阶段，无后端 API
- 状态管理集中在 App.jsx
- 使用 Tailwind CSS + 自定义语义色变量
- 需要保持与现有设计语言一致

## Goals / Non-Goals

**Goals:**

- 活动时间使用日历选择器，统一展示格式为 "4.28 周一 14:00"
- 扩展活动分类，添加"答辩"、"校招"、"实习招聘"等类型
- 管理员可选择活动海报在首页轮播展示
- 管理员可查看被举报帖子的完整内容
- 用户可创建收藏文件夹，收藏时选择文件夹
- 活动可以被收藏

**Non-Goals:**

- 后端 API 对接
- 移动端适配
- 收藏夹分享/协作
- 活动编辑功能
- 轮播图自动轮播

## Decisions

### 1. 日历时间选择器实现方案

**决策**：使用 HTML5 `<input type="datetime-local">` 配合自定义格式化函数。

**理由**：
- 原生控件无需引入额外依赖
- 浏览器原生支持日历选择，用户体验好
- 可以通过 JavaScript 格式化为中文友好的 "4.28 周一 14:00" 格式

**替代方案**：使用 react-datepicker 等第三方库 — 功能更强但增加 bundle 大小，当前阶段不值得。

### 2. 活动分类扩展策略

**决策**：在现有分类基础上新增"答辩"、"校招"、"实习招聘"、"校园招聘会"四个分类。

**理由**：这些是南京大学校园最重要的活动类型，用户需求高。

**数据结构**：
```javascript
const EVENT_CATEGORIES = [
  '官方活动', '学术讲座', '文艺展览', '体育赛事', '科技竞赛', '志愿公益',
  '答辩', '校招', '实习招聘', '校园招聘会' // 新增
];
```

### 3. 轮播管理数据结构

**决策**：在 localStorage 中维护 `nju_carousel_items` 数组，每个轮播项包含活动引用和显示顺序。

**理由**：轻量级实现，管理员可以添加/移除/排序轮播项，无需复杂后端。

**数据结构**：
```javascript
{
  id: 'carousel-1',
  type: 'event',
  eventId: 'PE-123456',
  order: 0,
  addedAt: '2026-04-24'
}
```

### 4. 举报详情查看方案

**决策**：在 AdminPage 中新增帖子详情弹窗（PostDetailModal），复用现有的帖子卡片组件。

**理由**：保持 UI 一致性，减少重复代码。现有帖子卡片已包含标题、内容、标签、情绪等信息。

### 5. 收藏文件夹架构

**决策**：收藏系统采用"文件夹 + 收藏项"两层结构。

- **文件夹**：用户创建的分类容器，包含名称、图标、创建时间
- **收藏项**：指向帖子/活动的引用，包含文件夹 ID、收藏时间

**数据结构**：
```javascript
// 文件夹
{
  id: 'folder-1',
  name: '课程相关',
  icon: 'book',
  createdAt: '2026-04-24'
}

// 收藏项
{
  id: 'bookmark-1',
  itemId: 'P-1234',  // 帖子或活动 ID
  itemType: 'post' | 'event',
  folderId: 'folder-1',  // 'all' 表示全部
  createdAt: '2026-04-24'
}
```

**理由**：简单直观，一个收藏项可以属于多个文件夹（通过 folderId 数组实现），但当前 MVP 阶段限制为单文件夹，简化实现。

### 6. 收藏文件夹默认行为

**决策**：系统预置"全部"文件夹（不可删除），所有收藏的内容自动出现在"全部"中。用户收藏时选择具体文件夹，不选则默认为"全部"。

**理由**：降低用户理解成本，"全部"作为兜底容器确保用户不会找不到收藏的内容。

### 7. 活动收藏实现

**决策**：活动收藏与帖子收藏使用相同的数据结构和 UI，通过 `itemType` 字段区分。

**理由**：代码复用，统一收藏体验。用户无需区分是收藏帖子还是活动，都是"收藏到文件夹"。

## Risks / Trade-offs

- **localStorage 容量**：收藏文件夹和轮播配置会增加存储使用 → 当前校园场景数据量有限，风险极低
- **数据格式兼容**：收藏数据结构变更需要迁移现有收藏 → 在 App.jsx 中做兼容处理，首次加载时将旧收藏数据迁移到新结构
- **时间格式化精度**：中文"星期x"需要正确计算 → 使用 JavaScript Date.toLocaleString('zh-CN', { weekday: 'short' }) 获取
- **轮播图手动管理**：需要管理员手动添加/移除 → 当前阶段可接受，后续可实现自动轮播和智能推荐
