## Context

当前 NJU 树洞前端已完成校园公告（AnnouncementsPage）和管理后台（AdminPage）的基础功能，但两者之间的流程存在断裂：

- 校园公告页面有详情弹窗，但管理后台审核队列中缺少类似能力
- 活动申请提交后存入 `pendingEvents` 状态，通过/拒绝只是从队列移除，没有状态追踪
- 通过审核的活动不会出现在校园公告页面
- 拒绝操作无理由输入，无反馈
- 管理后台整体布局需要重新设计，当前视觉效果差、信息层次混乱
- 活动过期后没有归档处理，会一直展示在公告列表中

技术约束：
- 当前为纯前端 mock 阶段，数据通过 localStorage 持久化
- 无后端 API，状态管理集中在 App.jsx
- 使用 Tailwind CSS + 自定义语义色变量

## Goals / Non-Goals

**Goals:**

- 管理员审核活动时可查看完整详情（复用公告详情弹窗的设计模式）
- 管理后台整体 UI 重新设计，视觉层次清晰，操作便捷
- 拒绝时必须填写理由，理由作为反馈通知给申请人
- 通过审核时发送通知（已通过审核 + 已在校园公告展示）
- 通过审核的活动自动出现在校园公告列表中
- 活动状态（待审核/已通过/已拒绝）可追踪
- 活动过期后归档到"历史活动"区域

**Non-Goals:**

- 活动编辑功能
- 活动报名/签到
- 后端 API 对接
- 移动端适配

## Decisions

### 1. 活动数据流：扩展状态管理

**决策**：在 App.jsx 中维护四个独立的 localStorage 键来管理活动生命周期：
- `nju_pending_events`：待审核活动（已存在）
- `nju_approved_events`：已通过活动（新增）
- `nju_rejected_events`：已拒绝活动，含拒绝理由（新增）
- `nju_archived_events`：已过期的历史活动（新增）

**理由**：相比将所有活动放在一个数组中用 status 字段过滤，独立键更简单，且与现有 `pendingEvents` 模式一致。避免修改现有数据结构，降低迁移风险。

**替代方案**：统一 `events` 数组 + status 过滤 — 更规范但需要修改现有逻辑，当前 mock 阶段不值得。

### 2. 审核详情弹窗：复用公告详情模式

**决策**：在 AdminPage 中新增活动详情弹窗，复用 AnnouncementsPage 的弹窗设计（modal-overlay + modal-content），展示完整活动信息（海报、描述、时间、地点、类型）。

**理由**：保持 UI 一致性，减少重复代码。公告详情弹窗已有成熟的样式和交互模式。

**替代方案**：跳转到公告页面查看 — 上下文切换，审核效率低。

### 3. 拒绝理由弹窗：预设理由 + 自定义输入

**决策**：点击拒绝后弹出理由弹窗，提供 5 个预设理由选项 + 自定义输入框，理由必填。

**理由**：预设理由降低输入成本，自定义输入覆盖边缘情况。必填确保申请人总能获得反馈。

### 4. 通过后发布：approvedEvents 合并到公告列表

**决策**：AnnouncementsPage 接收 `approvedEvents` prop，与 `SEED_ANNOUNCEMENTS` 合并渲染。approved 事件格式与 SEED_ANNOUNCEMENTS 统一。

**理由**：最小改动，不需要改公告卡片的渲染逻辑。通过审核时将 pendingEvent 转换为 announcement 格式即可。

### 5. 管理后台整体 UI 重设计

**决策**：使用 `impeccable` skill 重新设计整个 AdminPage 的布局和视觉风格，包括审核队列、举报队列、统计数据、侧边面板等所有区块。

**理由**：当前管理后台整体布局混乱，仅优化审核队列无法解决根本问题。整体重设计能确保各区块之间的视觉一致性和信息层次。

### 6. 活动过期归档

**决策**：活动过期后自动归档到"历史活动"区域。在校园公告页面底部展示"往期活动"折叠区域，归档活动在此展示。判定依据为活动时间是否已过（当前 mock 阶段可手动标记或通过时间字符串简单判断）。

**理由**：过期活动保留展示会干扰用户对当前活动信息的获取，归档到独立区域既保留历史又保持列表清爽。

**替代方案**：直接删除过期活动 — 信息丢失，用户可能需要回看。

### 7. 审核通知系统

**决策**：通过和拒绝都发送通知，复用 App.jsx 中现有的 `notifs` 状态。
- 通过通知：包含活动标题 + "已通过审核" + "已在校园公告展示"
- 拒绝通知：包含活动标题 + 拒绝理由

**理由**：审核是双向结果，申请人应知道审核进展，无论通过还是拒绝。

## Risks / Trade-offs

- **localStorage 容量**：大量活动可能撑满 localStorage → 当前校园场景活动量有限，风险极低
- **无实时通知**：通知仅存于前端 localStorage，不会真正推送 → 当前为 mock 阶段，后续对接后端时再实现推送
- **数据格式兼容**：approvedEvent 需要与 SEED_ANNOUNCEMENTS 格式对齐 → 在通过审核时做格式转换，确保 id/title/type/place/time/status/image 字段一致
- **UI 复杂度**：AdminPage 增加两个弹窗 + 整体重设计 → 通过组件拆分（ReviewDetailModal、RejectionModal）控制复杂度，使用 impeccable skill 保证设计质量
- **过期判断精度**：mock 阶段活动时间是字符串（如"4月28日 09:00"），难以精确判断 → 使用简单规则或手动归档，后续对接后端时用 Date 字段精确判断
