## Context

NJU树洞是一个校园匿名社区平台，当前使用 React + Vite（前端）和 Node.js + Express + MongoDB（后端）。项目已有基础的页面和组件架构，采用 Tailwind CSS 进行样式设计，整体风格为玻璃拟态。

当前平台缺少日常互动机制来培养用户习惯。每日运势功能旨在通过轻松娱乐的内容增加用户粘性和日活。

## Goals / Non-Goals

**Goals:**
- 实现每日打卡运势功能，用户每天只能打卡一次
- 打卡前显示日期，打卡后显示随机签文和宜忌建议
- 提供丰富的宜忌描述库，每次随机抽取多条
- 保持与现有玻璃拟态设计风格的一致性
- 初期采用前端硬编码存储，便于快速开发和迭代

**Non-Goals:**
- 不涉及真实占卜或预测功能
- 不涉及社交分享功能
- 不涉及付费内容
- 初期不涉及数据库存储（可后续迁移）

## Decisions

### 1. 打卡状态管理
**决策**: 使用 localStorage 存储用户打卡状态
**理由**: 
- 无需后端即可实现完整功能
- 降低初期开发复杂度
- 数据持久化在用户浏览器中
- 后期可无缝迁移到后端 API

**替代方案**: 后端 API 存储
- 优点: 跨设备同步、数据更安全
- 缺点: 增加开发复杂度、需要用户认证

### 2. 运势内容存储
**决策**: 初期使用前端静态数据，后期迁移至数据库
**理由**:
- 快速开发和内容迭代
- 便于 AB 测试不同的内容组合
- 内容量可控（宜忌描述各 50-100 条）

**迁移路径**: 当内容量超过 200 条或需要后台管理时，迁移至 MongoDB

### 3. 随机算法
**决策**: 使用日期作为种子进行伪随机
**理由**:
- 同一天内多次访问看到相同结果
- 用户体验一致
- 使用简单日期哈希: `hash(date + userId)`

### 4. UI 集成方式
**决策**: 作为独立卡片组件集成到首页
**理由**:
- 首页流量最大，曝光率高
- 不干扰现有功能
- 可独立开发和测试

### 5. 宜忌展示数量
**决策**: 每次显示 3 条宜、3 条忌
**理由**:
- 内容丰富但不冗长
- 参考图片中的布局比例
- 用户注意力集中在 3-5 条内容

## Risks / Trade-offs

### Risk 1: 内容枯竭
**风险**: 用户多次打卡后看到重复内容
**缓解**:
- 初期准备 50+ 条宜忌描述
- 使用日期+用户ID作为随机种子，不同用户每天看到不同内容
- 后期支持动态内容更新

### Risk 2: 跨设备体验不一致
**风险**: localStorage 数据无法跨设备同步
**缓解**:
- 明确告知用户本地存储特性
- 后期提供后端 API 选项

### Risk 3: 时区问题
**风险**: 不同时区用户打卡日期可能不一致
**缓解**:
- 使用用户本地时区判断日期
- 在午夜 00:00 重置打卡状态

## 数据结构设计

### localStorage 存储结构
```typescript
interface FortuneData {
  lastCheckInDate: string;      // 格式: YYYY-MM-DD
  fortune: {
    type: string;               // 签文类型（上上签/上签/中签/下签）
    text: string;               // 签文内容
    dos: string[];              // 今日宜（3条）
    donts: string[];            // 今日忌（3条）
  };
  history: FortuneHistory[];    // 历史记录
}

interface FortuneHistory {
  date: string;
  fortune: FortuneData['fortune'];
}
```

### 运势内容库结构
```typescript
interface FortuneContent {
  fortunes: {
    type: string;
    text: string;
  }[];
  dos: string[];      // 宜事项描述库
  donts: string[];    // 忌事项描述库
}
```

## Open Questions

1. **签到时间**: 是否需要设置每日签到时间窗口（如 00:00-23:59）？
   - 建议: 使用自然日（本地时区 00:00 重置）

2. **历史记录保留时长**: localStorage 中保留多少天的历史？
   - 建议: 保留最近 30 天

3. **是否需要重置功能**: 用户是否可以手动清除打卡数据重新抽取？
   - 建议: 提供"重新抽取"按钮（仅限测试或特殊需求）
