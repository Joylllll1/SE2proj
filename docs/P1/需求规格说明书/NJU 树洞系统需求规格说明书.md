# NJU 树洞系统需求规格说明书（SRS）

## 1. 引言

### 1.1 目的（Purpose）

本文档用于描述 NJU 树洞系统的功能需求与非功能需求，为系统设计、开发、测试与验收提供统一依据。
该系统是一个校园匿名UGC社区平台，支持用户在匿名环境下进行内容发布、互动与内容治理。

### 1.2 范围（Scope）

系统主要功能包括：
- 用户注册与登录
- 帖子发布与管理
- 匿名互动（评论、点赞、收藏）
- 内容搜索与标签系统
- 举报与审核机制
- 管理员治理功能
- 可选扩展功能（AI辅助、公告、运势等）

系统核心特点为：
- 帖子级匿名机制
- 可追责后台体系
- UGC驱动内容生态

### 1.3 定义（Definitions）

| 术语 | 定义 |
| --- | --- |
| user_id | 用户唯一身份标识（后台使用） |
| post_id | 帖子唯一标识 |
| comment_id | 评论唯一标识 |
| anon_id | 用户在某一帖子内的匿名标识 |
| SRS | Software Requirements Specification（需求规格说明书） |
| UGC | User Generated Content（用户生成内容） |

### 1.4 参考文献（References）

- IEEE Std 830-1998 Software Requirements Specification
- NJU 树洞系统需求讨论稿（AI + 人工融合版本）

## 2. 总体描述

### 2.1 产品视角（Product Perspective）

本系统为独立校园匿名社区平台，支持用户在匿名环境中进行内容发布与互动。
系统采用分层架构：
- 前端展示层（匿名展示）
- 后端服务层（身份与内容管理）
- 数据存储层（用户、帖子、评论、映射关系）

### 2.2 用户特征（User Classes）

| 用户类型 | 特征 |
| --- | --- |
| 普通用户 | 注册登录、发帖、评论、互动 |
| 管理员 | 内容审核、举报处理、用户管理 |
| 游客（可选） | 浏览部分公开内容 |

### 2.3 约束（Constraints）

- 必须支持匿名机制且不可跨帖关联
- 必须支持后台追责能力
- 用户身份数据必须加密存储
- 系统需支持基本并发访问需求

### 2.4 假设与依赖（Assumptions）

- 用户具备校园统一身份或邮箱注册能力
- 系统部署在稳定服务器环境
- 基础数据库服务可用（MySQL/PostgreSQL）

## 3. 功能需求（Functional Requirements）

### 3.1 用户与身份系统

#### FR-1 用户注册与登录

**描述：** 用户必须通过注册流程创建系统账号并完成身份认证后方可使用系统功能。系统支持基于校园统一身份认证或邮箱注册的方式（如果可以的话），并在登录后为用户生成唯一 user_id，用于后台身份管理与权限控制。系统需支持登录态保持与退出登录功能。
**优先级：** 高

**来源：** [AI生成 + 人工补充] 人工补充了登陆方式

#### FR-2 帖子级匿名机制

**描述：**
系统采用帖子级匿名机制，使用户在不同帖子中对应不同匿名身份。
- 在同一帖子内，用户的匿名身份保持一致，从而支持连续互动。
- 不同帖子之间的匿名身份完全隔离，无法建立关联。
- 该机制用于保证前端匿名性与后台可追溯性的平衡。

**匿名生成规则：**
anon_id 由 user_id 与 post_id 通过不可逆映射生成（如哈希或加密映射）。 
anon_id = hash(user_id + post_id)
**优先级：** 高
**来源：** [人工生成 + AI补充] AI补充了匿名机制

#### FR-3 匿名展示

**描述：**
系统前端仅展示匿名化后的用户标识（如 User01、Alice 等），不展示任何真实身份信息（如 user_id、邮箱或学号）。匿名昵称在同一帖子内保持稳定，用于维持讨论连续性，但不会跨帖子复用，以防止身份关联与行为追踪。
**优先级：** 高
**来源：** [人工生成 + AI补充] AI补充了匿名机制

#### FR-4 后台身份映射

**描述：**
系统在后台维护 user_id 与 anon_id 之间的映射关系，用于内容审计、违规追踪与安全管理。该映射仅限管理员或授权审计模块访问，不对普通用户开放，并需配合日志记录机制，确保所有身份查询行为可追踪。
**优先级：** 高
**来源：** [人工生成 + AI补充] AI补充了匿名机制

### 3.2 内容发布系统

#### FR-5 发帖功能

**描述：**
用户可创建新帖子并发布内容，支持文本与图片等。同时用户可为帖子添加一个或多个标签（tag），用于内容分类与检索优化。系统在发布时自动绑定当前帖子的 anon_id，并记录发布时间、内容版本及相关元数据。
**字段包括：**
- content
- media
- tag
- anon_id / timestamp / version metadata

**优先级：** 高
**来源：** [人工生成 + AI补充] AI补充了字段说明

#### FR-6 草稿功能

**描述：**
用户在未完成编辑或未准备发布时，可将帖子内容保存为草稿。草稿内容仅对用户本人可见，不参与公开展示或检索，并支持后续继续编辑、删除或一键发布。
**优先级：** 高
**来源：** [人工生成]

#### FR-7 帖子删除与修改

**描述：**
用户可对自己创建的帖子进行删除或修改操作。修改操作需记录版本变更信息（可选），以便系统维护一致性；删除操作需根据系统策略执行逻辑删除或物理删除，并同步更新相关索引与展示状态。
**优先级：** 高
**来源：** [人工生成]

### 3.3 互动系统

#### FR-8 评论系统

**描述：**
系统支持用户对帖子进行评论，并支持多级嵌套回复结构。评论需绑定对应帖子及 anon_id，使评论者在当前帖子语境下保持匿名一致性，同时支持评论内容的时间排序与层级展示。
**优先级：** 高
**来源：** [人工生成]

#### FR-9 点赞与收藏

**描述：**
用户可对帖子或评论进行点赞与收藏操作，用于表达态度或内容保存。系统需防止重复点赞，并记录用户与内容之间的交互关系，以支持热度计算与个性化推荐。
**优先级：** 中
**来源：** [人工生成]

#### FR-10 帖主管理评论

**描述：**
帖子创建者（帖主）拥有对其帖子下评论的管理权限，可删除违反规则或不当评论。该操作仅影响评论展示状态，不影响原帖内容，并需记录操作日志以供审计。
**优先级：** 中
**来源：** [人工生成]

### 3.4 内容治理系统

#### FR-11 举报功能

**描述：**
用户可对帖子、评论或其他内容进行举报，并选择举报类型（如违规、广告、辱骂等）。举报内容进入后台审核队列，由管理员统一处理。
**优先级：** 高
**来源：** [人工生成]

#### FR-12 管理员审核

**描述：**
管理员具备内容治理权限，包括：
- 审核举报内容并删除违规帖子或评论
- 对用户进行禁言或限制操作
- 所有管理行为需记录审计日志，并支持追溯操作历史

**优先级：** 中
**来源：** [AI生成 + 人工补充] 人工补充了管理员的具体权限

### 3.5 内容组织与分发

#### FR-13 排序系统

**支持：**
- 按发布时间排序（最新优先）
- 按互动热度排序（点赞/评论/收藏综合计算）
- 可扩展的混合排序策略，用于提升内容分发效率与用户体验

**优先级：** 中
**来源：** [AI生成]

#### FR-14 标签系统

**描述：**
系统支持为帖子添加标签，用于内容分类、筛选与推荐。标签可用于构建主题聚合页面，并辅助搜索与推荐算法提升内容发现效率。
**优先级：** 中
**来源：** [人工生成]

#### FR-15 搜索功能

**描述：**
系统支持基于 post_id 的精确搜索以及基于标签的条件过滤查询。后续可扩展支持关键词全文搜索，以提升内容检索能力与用户查找效率。
**优先级：** 高
**来源：** [AI生成 + 人工补充] 人工补充搜索方式

### 3.6 扩展功能（可选）

#### FR-16 公告系统

经管理员授权的用户可发布全站公告，用于活动通知或重要信息发布。公告内容在全站范围内可见，并具有较高展示优先级，不参与普通帖子排序机制。
**优先级：** 低
**来源：** [人工生成]

#### FR-17 AI辅助模块

**包括：**
- 内容摘要生成
- 违规内容初筛
- 基础对话辅助功能，用于提升内容处理效率与用户交互体验

**优先级：** 低
**来源：** [人工生成]

#### FR-18 每日运势功能

系统提供娱乐性功能模块，如每日运势、抽签或随机内容生成等，用于增强用户活跃度与平台趣味性，该模块不参与核心业务逻辑。
**优先级：** 低
**来源：** [人工生成]

## 4. 非功能需求（Non-functional Requirements）

### 4.1 性能需求（Performance）

- 系统支持高并发访问
- 页面响应时间 ≤ 2s
- 评论更新延迟可控（可选实时）

**来源：** [AI生成]

### 4.2 安全性需求（Security）

- user_id 必须加密存储
- anon_id 不可逆推 user_id
- 禁止跨帖子身份关联
- 管理员权限需认证
- 敏感操作需审计日志

**来源：** [AI生成]

### 4.3 可用性需求（Usability）

- UI 简洁易用
- 支持移动端访问
- 匿名机制对用户透明

**来源：** [AI生成]

### 4.4 可维护性需求（Maintainability）

- 模块化设计（身份/内容/治理分离）
- 支持功能扩展（如AI模块）
- 日志系统完整可追踪

**来源：** [AI生成]

### 4.5 可靠性需求（Reliability）

- 系统稳定运行
- 数据持久化存储
- 异常情况可恢复

**来源：** [AI生成]

本文档定义了 NJU 树洞系统的完整需求模型，包括匿名机制、内容系统、互动系统与治理机制。

- 可追责性（后台可审计）
- 可扩展性（支持AI与功能扩展）
- UGC社区生态基础能力
