# NJU树洞

> 面向南京大学师生的半匿名表达、互助交流与内容治理平台

## 项目概述

NJU树洞希望为校园用户提供一个更低压力、更有边界感的表达空间，用于情绪倾诉、日常分享、校内互助、问题求助与经验交流。

和传统社交平台相比，这个项目更关注三个问题：

- 熟人环境下“不想被看到，但又想说出来”的表达需求
- 匿名表达带来的安全感，与社区失控风险之间的平衡
- 面向校园场景的信息组织、反馈效率与基本治理能力

目前仓库处于 **需求分析完成、工程脚手架初始化** 阶段。`frontend/` 与 `backend/` 已建立基础 Node.js 项目结构，后续将围绕需求文档逐步实现功能。

## 为什么做这个项目

根据问卷、访谈、用户故事与核心用例整理，团队观察到几类比较一致的需求：

- 用户确实存在匿名表达情绪、吐槽、求建议和分享经历的场景
- 现有渠道如朋友圈、QQ 动态、博客或私聊，常见问题是熟人可见、反馈失衡、隐私边界模糊
- 一部分用户希望获得共鸣和互动，另一部分用户更希望“只记录、不被打扰”
- 用户普遍希望平台具备举报、审核、屏蔽恶意内容等治理能力

因此，NJU树洞不是单纯的“匿名发帖板”，而是一个强调 **匿名保护、低压表达、可治理** 的校园社区系统。

## 目标用户

- 普通用户：浏览、发帖、评论、点赞、收藏、搜索、保存草稿
- 管理员：审核举报、删除违规内容、禁言用户、查看身份映射
- 游客：浏览部分公开内容，了解社区氛围

## 典型使用场景

- 不想向熟人公开表达的情绪倾诉
- 对课程、校园生活、宿舍关系等话题的匿名讨论
- 寻求建议、经验交流或互助信息
- 仅记录当下心情，不希望被强互动打扰
- 通过标签和搜索快速发现相关话题

## 核心产品设计

### 1. 帖子级匿名

系统采用帖子级匿名机制：

- 同一用户在同一帖子中的匿名身份保持一致，便于连续讨论
- 同一用户在不同帖子中的匿名身份彼此隔离，降低跨帖关联风险
- 前台只展示匿名身份，后台保留受控映射关系用于审计与追责

这意味着产品定位不是“完全无约束匿名”，而是 **对外匿名、对内可治理**。

### 2. 低压力表达

结合访谈结果，平台设计会尽量兼顾两类需求：

- 想获得评论、安慰或共鸣的互动式表达
- 只想记录、不希望被打扰的轻互动表达

### 3. 内容治理

为了避免匿名社区常见的失控问题，系统规划包含：

- 举报机制
- 管理员审核
- 敏感内容拦截
- 评论管理与删除
- 审计日志与身份追溯能力

## 当前规划的功能范围

### Must

- 用户注册与登录
- 匿名发帖
- 浏览帖子详情
- 评论与点赞
- 删除或修改本人帖子

### Should

- 举报评论或帖子
- 帖主管理评论
- 管理员审核举报内容
- 搜索帖子
- 查看身份映射

### Could

- 游客浏览部分公开内容
- 草稿箱
- AI 辅助审核或内容总结
- 公告、轻娱乐模块等扩展功能
